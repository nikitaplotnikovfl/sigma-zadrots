# План работ и распределение по агентам

Связано с [SPEC.md](./SPEC.md).

## Архитектура репозитория (монорепо)

```
sigma_zadrots/
├─ docker-compose.yml        # postgres (+ при росте redis)
├─ packages/
│  ├─ shared/                # типы, zod-схемы FACEIT-ответов, общие утилиты
│  ├─ db/                    # Prisma schema, миграции, клиент
│  ├─ collector/             # FACEIT-клиент, нормализация, агрегатор, cron-воркер
│  ├─ api/                   # Fastify REST API
│  └─ web/                   # Vite + React фронтенд
└─ docs/                     # SPEC.md, PLAN.md
```

## Этапы (последовательность с барьерами)

### Этап 0 — Фундамент (блокирует всё)
- Монорепо (pnpm workspaces), TS-конфиги, lint/format, docker-compose с Postgres.
- `packages/shared`: zod-схемы ответов FACEIT + выводимые типы.
- `packages/db`: Prisma schema по разделу 3 ТЗ + первая миграция + seed-скрипт.
- **Выход:** компилируемое монорепо, поднятая БД, типы и схема готовы.

### Этап 1 — Сбор данных (параллелит часть работ)
- FACEIT API-клиент: bearer-auth, очередь с RPS-лимитом, retry/backoff на 429.
- Сборщик: список матчей источника → diff новых → `/matches/{id}/stats` → нормализация → upsert.
- Агрегатор: пересчёт `player_aggregate` (винрейт, K/D, ADR, HS%, rating v1).
- Cron-воркер + ручной триггер; журнал `sync_run`.
- **Выход:** по реальному источнику данные лежат в БД, агрегаты считаются.

### Этап 2 — API (зависит от 0 и 1)
- Эндпоинты лидерборда (сортировка/фильтр/пагинация на стороне БД), игрока, матчей, источников.
- Валидация query-параметров (zod), пагинация, обработка ошибок.
- **Выход:** API отдаёт данные, контракты задокументированы.

### Этап 3 — Фронтенд (контракты можно мокать параллельно со 2)
- Каркас приложения, роутинг, дизайн-система (Tailwind), TanStack Query-клиент.
- Лидерборд на TanStack Table (сортировка/фильтры/пагинация, серверная).
- Страница игрока (KPI + графики Recharts + история).
- Список матчей и страница матча.
- Страница источников (добавление по ссылке/ID, статус синка).
- **Выход:** рабочий UI поверх реального API.

### Этап 4 — Интеграция, деплой, полировка
- E2E-прогон на реальном турнире, фикс расхождений с FACEIT.
- Docker-образы api/collector, инструкция деплоя, прод-конфиг БД.
- README, переменные окружения, базовые тесты нормализации/агрегации.

## Распределение по агентам

Барьер после Этапа 0. Затем агенты A/B/C работают параллельно, синхронизируясь по
контрактам из `packages/shared` и Prisma-схемы.

| Агент | Зона ответственности | Этапы | Зависит от |
|---|---|---|---|
| **Agent-Foundation** | монорепо, tooling, docker, `shared` (zod-схемы), `db` (Prisma + миграции + seed) | 0 | — |
| **Agent-Collector** | FACEIT-клиент, троттлинг, сборщик, агрегатор, cron-воркер, `sync_run` | 1 | Foundation |
| **Agent-API** | Fastify, эндпоинты лидерборда/игрока/матчей/источников, валидация, пагинация | 2 | Foundation (+ Collector для данных) |
| **Agent-Web** | React-приложение, лидерборд-таблица, страница игрока, матчи, источники, графики | 3 | Foundation (контракты); мок API на старте |
| **Agent-Integration** | E2E на реальном турнире, деплой (docker), README, тесты | 4 | A/B/C |

Каждый агент работает в своём пакете → конфликтов по файлам почти нет. Контракт между
Collector→API→Web фиксируется типами в `packages/shared` и Prisma Client из `packages/db`.

## Порядок запуска

1. Сначала только **Agent-Foundation** (барьер — остальные ждут схему и типы).
2. Затем параллельно **Collector**, **API**, **Web** (Web стартует на моках).
3. Финал — **Agent-Integration** после готовности A/B/C.

## Что нужно от заказчика для старта Этапа 1

- FACEIT API key (server-side) и ссылки на хабы/чемпионаты (см. открытые вопросы в ТЗ).
- До получения ключа Foundation/каркас Collector/API/Web можно делать на фикстурах
  (сохранённых примерах JSON-ответов FACEIT).
