# ТЗ: Статистика турниров FACEIT Hub по CS2

Версия: 0.1 (черновик) · Дата: 2026-05-30

## 1. Цель

Веб-приложение, которое собирает данные по матчам турниров (хабов/чемпионатов)
FACEIT в CS2, агрегирует статистику по игрокам и показывает:

- **Лидерборд игроков** с сортировкой и фильтрацией;
- **Детальную страницу игрока** с историей и графиками;
- **Список матчей** турнира с подробной статистикой по каждой игре;
- **Авто-обновление** данных по расписанию.

## 2. Источник данных — FACEIT Data API

- Base URL: `https://open.faceit.com/data/v4`
- Авторизация: server-side API key, заголовок `Authorization: Bearer <FACEIT_API_KEY>`
  (ключ выпускается на https://developers.faceit.com → создать приложение → Server-side API key).
- Формат: REST + JSON. Лимиты: возможен `429 Too Many Requests` — обязательно
  троттлить запросы (очередь с ограничением RPS + backoff на 429).

### Используемые эндпоинты

| Назначение | Эндпоинт |
|---|---|
| Инфо о хабе | `GET /hubs/{hub_id}` |
| Матчи хаба (постранично) | `GET /hubs/{hub_id}/matches?type=past&offset=&limit=` |
| Участники хаба | `GET /hubs/{hub_id}/members` |
| Инфо о чемпионате | `GET /championships/{championship_id}` |
| Матчи чемпионата | `GET /championships/{championship_id}/matches` |
| Детали матча | `GET /matches/{match_id}` |
| **Статистика матча** | `GET /matches/{match_id}/stats` |
| Профиль игрока | `GET /players/{player_id}` |

> Турнир на FACEIT может быть оформлен как **Hub** или как **Championship** —
> поддерживаем оба типа как «источник» (source).
>
> **Важно (выяснено на практике):** ссылка вида `faceit.com/ru/club/{guid}` —
> это FACEIT 2.0 **Club**, и `{guid}` совпадает с `hub_id`. НО публичный Data API
> отдаёт хаб только если он **опубликован** (`published: true`). Непубличный хаб
> возвращает 404 на `/hubs/{id}` и пустой список на `/matches`. Поэтому для сбора
> данных хаб должен быть опубликован организатором (внутренний API
> `www.faceit.com/api/hubs/v1/hub/{id}` показывает инфо даже у приватного, но список
> матчей у приватного требует авторизованной сессии — это не наш путь).
>
> **Важно №2 (выяснено при сборе):** id из ссылки `/club/0dd077bc-...` — это обёртка-**клуб**,
> у которого в Data API 0 матчей. Реальная **hub-очередь** клуба (где играются матчи) — отдельный
> id `d0701937-8eba-4df9-8830-22137001c0bd` (хаб «5vs5», тот же организатор `nycujan`).
> Нашли его через историю матчей участника: `GET /players/{member}/history?game=cs2` →
> поле `competition_id` у матчей с `competition_type: hub`. Это рабочий источник: 38 матчей.
>
> Текущий источник (`HUB_ID` в `packages/server/.env`): `d0701937-8eba-4df9-8830-22137001c0bd`.
> Клуб `0dd077bc-...` хранится как `CLUB_ID` для брендинга.

> Локальный dev: Vite-прокси `/api` указывает на `http://127.0.0.1:3000` (не `localhost`),
> иначе Node 18 резолвит в IPv6 `::1` и прокси падает с 500 (API слушает IPv4).

### Структура `GET /matches/{match_id}/stats` (CS2)

`rounds[]` → каждый раунд (карта) содержит `round_stats` (Map, Score, Winner, Rounds)
и `teams[]`; у каждой команды `team_stats` и `players[]`; у игрока `player_stats`:
`Kills, Deaths, Assists, K/D Ratio, K/R Ratio, Headshots, Headshots %, MVPs, ADR,
Triple/Quadro/Penta Kills, Result` (+ для CS2 — урон, утилити, флешки, entry, если доступны).

## 3. Модель данных (PostgreSQL)

- `source` — отслеживаемый турнир: `id, type (hub|championship), faceit_id, name, game, added_at`.
- `match` — `id (faceit match_id), source_id, status, best_of, started_at, finished_at, winner_team, raw_meta`.
- `match_map` — карта внутри матча: `id, match_id, map_name, rounds, score, winner_team`.
- `team` — `id (faceit team_id), match_id, name, final_score, is_winner`.
- `player` — справочник: `id (faceit player_id), nickname, avatar, country, faceit_url`.
- `player_match_stats` — строка статистики игрока за карту/матч: `match_id, map_id, player_id, team_id,
  kills, deaths, assists, kd, kr, hs, hs_pct, adr, mvps, triple/quadro/penta, damage, result(win)`.
- `player_aggregate` (материализованное представление или таблица-кэш) — агрегаты на игрока
  для лидерборда: матчи, винрейт, сумм/средние kills/deaths/adr/hs%, K/D, K/R, рейтинг (HLTV-подобный), и т.д.
- `sync_run` — журнал прогонов сборщика: `started_at, finished_at, source_id, matches_seen, matches_new, status, error`.

Агрегаты пересчитываются после каждого прогона сборщика (инкрементально либо полным
пересчётом для затронутых игроков).

### Метрики лидерборда (минимум)

Матчи, Винрейт %, Всего килов/смертей/ассистов, K/D, K/R, ADR, HS%, MVPs,
средний Kills/матч, многоубийства (3k/4k/5k), и производный **Rating** (формула на базе
K/R, KAST-приближения, ADR — фиксируем v1, документируем).

## 4. Бэкенд (Node + TypeScript)

- **API-сервер**: Fastify (TS). REST-эндпоинты:
  - `GET /api/leaderboard?source=&sort=&order=&filter[...]=&minMatches=&page=`
  - `GET /api/players/:id` — агрегаты + история матчей + точки для графиков
  - `GET /api/matches?source=&page=` и `GET /api/matches/:id` — детали + по-картовая стата
  - `GET /api/sources` — список отслеживаемых турниров; `POST /api/sources` — добавить по ссылке/ID
- **Сборщик (worker)**: отдельный процесс. Шаги: получить матчи источника → определить новые →
  для каждого нового тянуть `/matches/{id}/stats` → нормализовать → upsert в БД → пересчёт агрегатов.
  Троттлинг через очередь (p-queue) с лимитом RPS и экспоненциальным backoff на 429.
- **Расписание**: node-cron внутри воркера (интервал из конфига, напр. каждые 10 мин) +
  ручной триггер `POST /api/sync` (защищённый).
- **ORM**: Prisma (миграции, типобезопасность).
- **Конфиг**: `.env` — `FACEIT_API_KEY`, `DATABASE_URL`, `SYNC_INTERVAL`, `SYNC_RPS`.

## 5. Фронтенд (React + TypeScript)

- Vite + React + TS, **TanStack Query** (загрузка/кэш), **TanStack Table**
  (сортировка/фильтрация/пагинация лидерборда), **Tailwind** (стили), **Recharts** (графики).
- Страницы:
  - **Лидерборд** — таблица, сортировка по любой колонке, фильтры (мин. матчей, страна, источник,
    поиск по нику), переключатель метрик, пагинация.
  - **Игрок** — карточка профиля, KPI-плитки, графики динамики (K/D, ADR по времени),
    таблица истории матчей.
  - **Матчи** — список матчей источника; **Матч** — скор по картам, составы, по-игроковая стата.
  - **Источники** — добавление/просмотр отслеживаемых турниров, статус последнего синка.

## 6. Хранилище и деплой (решение принято)

- **БД**: PostgreSQL — прод-таргет (Neon/Supabase/Railway), переключение через `DATABASE_URL`.
  **Локально сейчас** — SQLite (`file:./dev.db`), т.к. Docker на машине недоступен; Prisma
  меняет провайдер сменой одной строки в `schema.prisma`.
- **Бэкенд (MVP)**: для скорости консолидирован в один пакет `packages/server`
  (Prisma + FACEIT-клиент + сборщик + агрегатор + Fastify API + cron). Разнесение на
  `db`/`collector`/`api` из PLAN — при росте.
- **Деплой**: фронт — статика (Vercel/любой静态-хостинг); API + worker — контейнеры
  (Railway/Fly.io/собственный VPS через docker-compose). Redis в MVP не нужен —
  планировщик и троттлинг работают в процессе воркера; добавим Redis+BullMQ при росте нагрузки.

## 7. Нефункциональные требования

- Идемпотентность сборщика (повторный запуск не дублирует данные — upsert по ключам).
- Устойчивость к 429/таймаутам (retry + backoff, журнал в `sync_run`).
- Все ключи и секреты — только на бэкенде, фронт ходит только в наш API.
- Кэш агрегатов, чтобы лидерборд отдавался быстро при тысячах матчей.

## 8. Решения по объёму (закрыто с заказчиком)

1. **Источник:** один FACEIT **Hub**, все его матчи. Переключатель источников в UI не нужен,
   но модель данных оставляем мульти-источниковой (на будущее).
2. **Глубина истории:** тянем **все прошлые матчи** хаба с самого начала
   (постраничный обход `/hubs/{hub_id}/matches?type=past`). Первичный сбор — полный бэкафилл.
3. **Доступ:** лидерборд и вся статистика **публичны** (без логина). Админ-ручки
   (добавить источник, ручной синк) защищены **простым admin-токеном** (`ADMIN_TOKEN` в env,
   заголовок `X-Admin-Token`).
4. **Рейтинг игрока:** дефолтная формула v1 на базе K/R, ADR и приближённого KAST;
   фиксируется и документируется в `collector` (можно уточнить позже).

### Осталось получить от заказчика для старта реального сбора
- **FACEIT server-side API key** (developers.faceit.com → My Apps).
- **Ссылка на Hub** (из неё извлекаем `hub_id`).
- До этого: Этап 0 и каркасы делаем на сохранённых JSON-фикстурах.
